# Context for Claude Code

This file exists to carry context across machines/sessions. See [README.md](README.md)
for the project's purpose and scope — read that first. This file itself is
gitignored (not pushed to the public repo) since it's local working notes,
not documentation for other contributors.

## Status as of 2026-07-18

The v0.1 engine described in the README is implemented and verified —
`cargo build`, `cargo test` (11 tests), and `cargo clippy` are all clean.
This was not true earlier in the day; see "History" below for what it
took to get here.

### What's in place
- [proto/frame.proto](proto/frame.proto) — final schema: `CameraPose`,
  `CameraIntrinsics`, `Detection`, `Frame`, `MapObject`, `SceneMap`.
- [build.rs](build.rs) — compiles the proto via `prost-build`, with
  `type_attribute` adding `#[derive(serde::Serialize, serde::Deserialize)]`
  + `#[serde(default)]` to every generated type, so `Frame` can be loaded
  straight from JSON fixtures.
- [src/projection.rs](src/projection.rs) — ground-plane ray-cast
  projection (bbox center + pose + intrinsics → world (x, y)).
- [src/engine.rs](src/engine.rs) — `Engine::ingest_frame` does real
  association (nearest same-label object within a radius), fusion
  (running-average position, evidence-combined confidence), and map
  maintenance (single staleness rule prunes noise/stale/moved objects).
  `EngineConfig` holds the two tunable thresholds; see its doc comments
  for how `association_radius_meters` (0.3) was derived from a noise
  sensitivity analysis, vs. `stale_timeout_seconds` (2.0) which is a
  documented assumption, not empirically derived — no real detector
  dropout data exists yet to tune it from.
- [src/geojson.rs](src/geojson.rs) — `SceneMap::to_geojson()` /
  `to_geojson_json()` for output testable via matplotlib/geopandas.
- [src/fixture.rs](src/fixture.rs) — `replay_fixture()` loads a JSON
  array of `Frame` and replays it through an `Engine`, returning a
  `SceneMap` snapshot per frame.
- [tests/fixtures/session.json](tests/fixtures/session.json) — small
  hand-written fixture ([tests/replay.rs](tests/replay.rs)).
- [tests/fixtures/tum_freiburg1_xyz.json](tests/fixtures/tum_freiburg1_xyz.json) —
  real camera trajectory from the TUM RGB-D benchmark (CC BY 4.0) paired
  with synthetic detections (incl. deterministic +/-8px detector noise);
  generated by [scripts/gen_tum_fixture.py](scripts/gen_tum_fixture.py),
  exercised by [tests/replay_tum_fixture.rs](tests/replay_tum_fixture.rs).
- [LICENSE](LICENSE) — MIT, matching `Cargo.toml`.

### What's not done yet
- No CI workflow (nothing runs build/test/clippy automatically on push/PR).
- `stale_timeout_seconds` default is a reasoned guess, not validated —
  revisit once there's a real frontend/detector providing dropout data.
- Everything the README lists under "Longer-term (not v0.1)": depth-aware
  projection, visual re-identification, moving-object handling,
  multi-session map merging. These are intentionally out of scope, not
  overlooked.

### History (why the build was ever in question)
Earlier the same day, on a prior Windows machine, `cargo build` had never
successfully completed — not because of the Rust code, but environment
issues (stale VC98 linker on PATH, then missing MinGW `dlltool.exe`, then
a restricted network that couldn't fetch VS Build Tools). Work moved to
this machine instead. Here, on a fresh macOS environment, two different
(unrelated) gaps showed up and were fixed: the system rustc (1.81.0) was
too old for a transitive dependency needing `edition2024` (`rustup update
stable` → 1.97.1 fixed it), and `protoc` wasn't installed (`brew install
protobuf` fixed it). Neither of these will necessarily apply on yet
another machine — if `cargo build` fails, check the actual error rather
than assuming it's one of these.

### Git remote
`origin` is `git@github.com:ghimirebibek/geomap-engine.git`. This
sandbox's `git fetch`/`git push` fail with `Permission denied (publickey)`
— it doesn't have the user's SSH key. The user pushes from their own
machine/terminal after commits land here. Check `git status` /
`git log --oneline -1` vs. what's actually on GitHub before assuming
they're in sync; `git status` reporting "up to date with origin/main"
only reflects the last successful push from *somewhere*, not a live
check from this sandbox.
